# სახლის ფასების წინასწარმეტყველება

## Kaggle-ის კონკურსის მოკლე მიმოხილვა

ეს პროექტი მუშაობს Kaggle-ის კონკურსზე [House Prices - Advanced Regression Techniques](https://www.kaggle.com/competitions/house-prices-advanced-regression-techniques). მონაცემები შეიცავს 1460 სახლის ინფორმაციას, სადაც 79 feature აღწერს სახლის სხვადასხვა მახასიათებელს: ზომას, ხარისხს, ლოკაციას, ასაკს და სხვა. target ცვლადი არის `SalePrice`, ანუ საბოლოო გასაყიდი ფასი, რომელიც უნდა ვიწინასწარმეტყველოთ test set-ზე.

პროექტის მიზანი არ არის მხოლოდ საუკეთესო leaderboard ქულის მიღება. უფრო მნიშვნელოვანია, რომ სხვადასხვა მიდგომა გავტესტოთ cleaning-ის, feature engineering-ის, feature selection-ისა და მოდელირების ეტაპებზე, და გავიგოთ რა მუშაობს და რატომ.

## მიდგომა პრობლემის გადასაჭრელად

თავდაპირველად მონაცემებს შეუცვლელად დავაკვირდით, გავიგეთ რამდენიმე მნიშვნელოვანი ფაქტი მათ სესახებ რაც მათ გარდაქმნაში და დამუშავებაში დაგვეხმარებოდა, შემდგომ დავამუშავეთ ისინი რათა ყველა არარიცხვითი მნიშვნელობა რიცხვითად გადაგვექცია, feature engineering-ის ნაწილში დავამატეთ რამდენიმე თვისება, რომელიც შესაძლოა მოდელისთვის მნიშვნელოვანი ყოფილიყო, გამოვიყენეთ რამდენიმე feature selection-ის ტექნიკა, და ყოველი მათგანის მიერ მიღებულ მონაცემებზე გავატარეთ 2 linear და 2 non-linear ტიპის მოდელი, Ridge, Lasso, Decision Tree, Random Forest. შემდგომ დავაკვირდით ყოველი მათგანის შედეგებს, ისინი შევადარეთ და ამოვარჩიეთ საუკეთესო შედეგის მქონე მოდელი.

## რეპოზიტორიის სტრუქტურა

* **`model_experiment.ipynb`** ძირითადი notebook, სადაც ხდება ყველა ექსპერიმენტი. შეიცავს EDA-ს, cleaning-ს, feature engineering-ს, feature selection-ს, training-ს და MLflow logging-ს. აქვე ხდება საუკეთესო მოდელის არჩევა და Model Registry-ში დარეგისტრირება.
* **`model_inference.ipynb`** test set-ზე პროგნოზის გაკეთება. Model Registry-დან იტვირთება საუკეთესო დარეგისტრირებული მოდელი და გენერირდება `submission.csv` ფაილი, რომელიც Kaggle-ზე აიტვირთა.
* **`README.md`** მოცემული ფაილი, სადაც დეტალურად არის აღწერილი ყველა ეტაპი და მიდგომა.

## EDA

სანამ მონაცემებს შევცვლიდი, უბრალოდ დავაკვირდი მათ, რამდენი ცვლადი გვქონდა სულ, რამდენი მონაცემი იყო ცარიელი, რამდენი ცვლადი იყო კატეგორიული და რამდენი რიცხვითო, რა განაწილება ჰქონდა გასაყიდ ფასებსა და მათ ლოგარითმებს, რა სახის outlier-ები გვქონდა. ამ ყველაფერმა წარმოდგენა შემიქმნა, თუ რა სახის მონაცემებთან მქონდა საქმე.

## Data Separation

საწყისი მონაცემები 80/20 შეფარდებით, შემთხვევითად გავყავი train set-ად და validation set-ად ფიქსირებული სიდით.

## Feature Engineering

### Cleaning მიდგომები

ამ ნაწილში დავაკვირდი train set-ს და შევეცადე ყველა მონაცემი რიცხვითად მექცია. პირველ რიგში, დატასეტიდან ამოვშალე outlier-ები.

### Nan მნიშვნელობების დამუშავება

შემდგომ დავაკვირდი ცარიელ მონაცემებს, და აღმოვაჩინე რამდენიმე ცვლადი, რომლის სიცარიელეც ინფორმაციის მომტანი იყო და უბრალოდ ამა თუ იმ თვისების არ არსებობას ნიშნავდა, მაგალითად ცვლადი, რომელიც აუზის ქონას განსაზღვრავს. ასეთი რიცხვითი ცვლადები 0-ებად გადავაქციე, ხოლო კატეგორიული ცვლადები "None" სტრინგებად. ასევე იყო ცვლადები, რომლის სიცარიელეც უბრალოდ მონაცემების არ არსებობით იყო გამოწვეული, მათთვის კი რეპრეზენტატორად მედიანა გამოვიყენე.

### კატეგორიული ცვლადების რიცხვითში გადაყვანა

შემდგომ, როცა ცარიელი ცვლადები აღარ მქონდა, გადავედი კატეგორიული ცვლადების რიცხვითით შეცვლაზე. შევამჩნიე, რომ ზოგიერთი ცვლადის შესაძლო მნიშვნელობების იერარქიულად დალაგება შესაძლებელი იყო, ამიტომ მათ მივანიჭე რიცხვები დალაგების მიხედვით, 0, 1, 2 და ასე შემდეგ. ასევე იყო ცვლადები, რომლის მნიშვნელობების იერარქიულად დალაგება შეუძლებელი იყო, მათთვის კი one-hot encoding გამოვიყენე. ამის შემდგომ დატასეტში არარიცხვითი მნიშვნელობები აღარ მქონდა.

### feature-ების დამატება

შემოვიღე რამდენიმე დამატებითი ცვლადი, რომელიც უკვე არსებული ცვლადების მანიპულაციით იყო მიღებული, და შესაძლოა გამოსადეგარი ყოფილიყო მოდელის დატრენინგებისას, მაგალითად სრული საცხოვრებელი ფართობი რომელიც საცხოვრებელი ფართობების ჯამით იყო მიღებული, ასაკი, რომელიც სახლის აშენების და გაყიდვის დროით იყო მიღებული და ა.შ. ამ ყველაფრის შემდეგ, ცვლადების რაოდებობა საკმაოდ გაიზარდა. გაზრდა გამოიწვია ჩემს მიერ დამატებულმა თვისებებმა და one-hot encoding ტექნიკამ, და მათი რაოდენობა 79-დან 231-მდე ავიდა. 

## Feature Selection

ამ ეტაპზე ცვლადების რაოდენობაზე მანიპულაციისთვის გამოვიყენე რამდენიმე Feature Selection ალგორითმი, და ყველა მათგანის მიერ მიღებული შედეგი შევინახე, რათა საბოლოოდ გამეგო, რომელმა მათგანმა იმუშავა ყველაზე მეტად. ასევე შევინახე ყველა ცვლადის მქონე მონაცემები baseline დატასეტად. ჩემს მიერ გამოყენებული მეთოდები ამ ეტაპზე იყო Variance Threshold, Correlation with Target, Correlation between Features, RFE (Recursive Feature Elimination), Tree-based Feature Importance, Lasso (L1 Regularization).

რამდენიმე მეთოდის გამოყენების მიზეზი ისაა, რომ თითოეულ მათგანს თავისი bias აქვს. მაგალითად Correlation with Target, RFE და Lasso უკეთ მუშაობენ წრფივი კავშირების აღმოჩენაში და შესაბამისად წრფივი მოდელებისთვის სასარგებლო feature-ებს ირჩევენ, ხოლო Tree-based Feature Importance უკეთ ხედავს nonlinear კავშირებს.

ყველა selection მეთოდი გამოვთვალე **მხოლოდ** `X_train`-ზე, რათა validation set-ის ინფორმაციას არ ეთამაშა როლი feature-ების არჩევის პროცესშის.

Baseline-ის გარდა, სხვა მეთოდებმა დატოვა: Variance Threshold 183, Correlation with Target 133, Correlation between Features 218, RFE 80, Tree-based Importance 100, Lasso ~90 ცვლადი.

## Training

### ტესტირებული მოდელები

დავტესტე 4 მოდელი, თითოეული განსხვავებული მიზნით:

* **Ridge** (L2 regularization) წრფივი მოდელი, რომელიც კარგად უძლებს multicollinearity-ს.
* **Lasso** (L1 regularization) წრფივი მოდელი, რომელიც თავისით აკეთებს implicit feature selection-ს, რადგან უსარგებლო ცვლადების კოეფიციენტებს ნულამდე ამცირებს.
* **Decision Tree** არაწრფივი მოდელი. ამას განზრახ ვტესტავდი unconstrained ვერსიით (max_depth გარეშე), რადგან ვიცოდი რომ ოვერფიტი იქნებოდა და მინდოდა მენახა რეალური მაგალითი.
* **Random Forest** (300 trees) bagging ensemble, რომელიც ბევრი decision tree-ის გასაშუალოებით თავიდან იცილებს ერთი ხის ოვერფიტინგს.

თითოეული მოდელი გავუშვი **ყველა feature set-ზე**, რამაც ჯამში 28 ექსპერიმენტი მომცა. ყველა ექსპერიმენტი დავლოგე, პარამეტრებით (model, feature_set, hyperparameters) და მეტრიკებით (train_rmse, val_rmse, overfit_gap).

### Hyperparameter ოპტიმიზაციის მიდგომა

ამ ეტაპზე მოდელების შედარების მიზანი იყო, რომ მენახა, რომელი კომბინაცია (მოდელი + feature set) იყო ყველაზე პერსპექტიული. ამიტომ გამოვიყენე default hyperparameter-ები თითოეული მოდელისთვის (Ridge-ზე alpha=1.0, Lasso-ზე alpha=0.001, Random Forest-ზე 300 ხე).

შედეგების მიხედვით, Lasso + corr_target კომბინაცია აღმოჩნდა საუკეთესო, val RMSE 0.1196, თუმცა სხვა კომბინაციებზე დაკვირვებამ საინტერესო შედეგები გვაჩვენა:

1) წრფივმა მოდელებმა უკეთესი შედეგი გვაჩვენა, ვიდრე არაწრფივმა. ლოგარითმულ სივრცეში მონაცემები საკმაოდ წრფივად განლაგებული აღმოჩნდნენ. ამიტომ იყო გასაყიდი ფასების ლოგარითმების აღება მნიშვნელოვანი. 

2) Decision Tree მოდელმა ოვერფიტის კარგი მაგალითი მოგვცა. ყოველი გაშვებისას, train RMSE იყო 0, რაც ნიშნავს, რომ ხემ სრულიად დაიზეპირა training data. val RMSE იყო 0.18-ზე მეტი, ანუ ხემ შეისწავლა ნოიზი სიგნალების ნაცვლად. ეს კარგად გვაჩვენებს, რომ unconstrainted ვერსია იქამდე იყოფა, სანამ ყოველი ფოთოლი ერთ წერტილს არ შეესაბამება, ხოლო მისი შედეგი არაა განზოგადებადი.

3) Random Forest მოდელმა აღმოფხვრა ეს მინუსი. Validation RMSE 0.18-დან 0.14-მდე შემცირდა, ხოლო Train RMSE 0.00-დან 0.05-მდე გაიზარდა, შესაბამისად, სხვაობა 0.18-დან 0.09-მდე შემცირდა. ბევრ სხვადასხვა Bootstrap ნიმუშზე დატრენინგებული ხეების პროგნოზების გასაშუალოებით, თითოეული ხის მიერ დამახსოვრებული ნოიზი იფილტრება, და რჩება სიგნალები.

4) corr_target საუკეთესო იყო წრფივი მოდელებისთვის: ეს ლოგიკურია, რადგან კორელაციის ფილტრი ეძებს ზუსტად იმ წრფივ კავშირებს, რომელთა გამოყენებაც Ridge-სა და Lasso-ს შეუძლიათ.


Lasso + corr_target კომბინაცია საუკეთესო იყო ორივე კრიტერიუმით: ყველაზე დაბალი val RMSE (0.1196) და ყველაზე პატარა overfit gap (+0.020). პატარა gap-ი თითქმის ისეთივე მნიშვნელოვანია, როგორც დაბალი error, რადგან ნიშნავს, რომ მოდელი კარგად განზოგადებადია. ამიტომ ავარჩიე ეს კომბინაცია საბოლოო პასუხად.

### ჩაწერილი მეტრიკების აღწერა

თითოეული run-ისთვის MLflow-ში დავლოგე შემდეგი ინფორმაცია:

* **Parameters:** `model` (მოდელის სახელი), `feature_set` (გამოყენებული selection მეთოდი), `n_features` (feature-ების რაოდენობა), `needs_scaling`, და ყველა hyperparameter.

* **Artifacts:** სრული sklearn Pipeline (scaler + model).
